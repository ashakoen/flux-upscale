# Flux Upscale — Assumptions & Build Plan

A Python CLI tool for quality, identity-preserving 2x–3x upscaling of Flux-generated images that used a subject LoRA. Designed to run on a RunPod GPU pod.

---

## Goals

- Single-command CLI: clone repo on a fresh RunPod box, install, run.
- Take a Flux-generated input image, produce a 2x or 3x upscaled version.
- Preserve subject identity using the same subject LoRA that was used to generate the source image.
- No ComfyUI. No graph UI. Just Python.
- Sensible defaults; named presets; flags for overrides.

---

## Assumptions

### Source material
- Input images are generated by **Flux** (presumed Flux.1-dev based on AI-Toolkit defaults).
- All inputs were generated using a **subject LoRA** trained via **ai-toolkit on Replicate** (Ostris's `ai-toolkit`).
- LoRAs are in `.safetensors` format with diffusers-compatible keys — `pipe.load_lora_weights()` should work without key remapping.
- Inputs are typically portrait or person-centric images.
- Upscale targets are **modest** — 2x or 3x, not 4x+.

### Hardware target
- **RunPod GPU pod** with H100, H200, or B200 (80GB+ VRAM).
- Full Flux.1-dev fits comfortably in fp16/bf16. No quantization, no CPU offload, no memory tricks needed.
- Persistent storage at `/workspace` on RunPod for model cache.

### Pipeline philosophy
- **Two-stage core**: conservative base upscale → low-denoise Flux img2img refinement with subject LoRA.
- **Identity preservation is the primary objective.** Detail enhancement is secondary.
- **Low denoise wins.** Default refinement strength stays low (0.10–0.20 range).
- **Reduced LoRA strength during upscale** vs. original generation strength — the LoRA acts as a stabilizer, not a creative driver.
- **Prompt is restraint-oriented**, not glamor-oriented.

### Out of scope for v1
- Face-only refinement pass (insightface crop → refine → composite). Defer until v1 is validated.
- Identity drift metrics (ArcFace embeddings). Useful for A/B testing later, not needed for v1.
- Batch processing. v1 is single-image; batch is a trivial wrapper later.
- Web UI / API server. CLI only.
- Non-Flux base models. SUPIR / SDXL refinement was considered and rejected — wrong family for Flux+LoRA sources.

## Pipeline Design

```
INPUT (Flux-generated image, e.g., 1024px)
    │
    ▼
┌─────────────────────────────────────────────┐
│ Stage 1: Conservative Base Upscale          │
│  - Real-ESRGAN (default) or SwinIR          │
│  - Pure non-generative — no hallucination   │
│  - Target: 2x or 3x of input                │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ Stage 2: Flux img2img Refinement            │
│  - Flux.1-dev with subject LoRA loaded      │
│  - Low denoise (0.10–0.20)                  │
│  - Reduced LoRA scale (~0.4–0.6 of original)│
│  - Restraint prompt + trigger token         │
└─────────────────────────────────────────────┘
    │
    ▼
OUTPUT (upscaled, identity-preserved)
```

## Presets

Three presets, all using the same two-stage pipeline with different parameter choices:

### `identity` — Identity First
For portraits where likeness matters most. Most conservative.
- Base upscaler: SwinIR
- Refinement denoise: **0.10**
- LoRA scale: **0.3**
- Guidance scale: low

### `balanced` — Balanced (default)
Everyday preset. Good detail without identity risk.
- Base upscaler: Real-ESRGAN
- Refinement denoise: **0.15**
- LoRA scale: **0.45**
- Guidance scale: moderate

### `detail` — Detail Push
For softer sources that need more help. Identity risk is higher — verify output.
- Base upscaler: Real-ESRGAN
- Refinement denoise: **0.22**
- LoRA scale: **0.6**
- Guidance scale: moderate

All values are **starting points** — meant to be tuned per LoRA via the A/B/C grid test later.

## CLI Surface

```
python upscale.py \
  --input path/to/source.png \
  --output path/to/output.png \
  --lora path/to/subject.safetensors \
  [--preset {identity,balanced,detail}] \
  [--scale {2,3}] \
  [--denoise FLOAT] \
  [--lora-scale FLOAT] \
  [--base-upscaler {realesrgan,swinir}] \
  [--prompt "trigger_token, additional restraint phrases"] \
  [--seed INT] \
  [--steps INT]
```

**Behavior:**
- `--preset` sets all the knobs; individual flags override preset values.
- `--prompt` is optional. If omitted, uses a default restraint prompt. Trigger token must be passed explicitly if the LoRA requires one.
- `--seed` is optional; random if omitted.

## Stack & Dependencies

### Core
- **`diffusers`** — Flux pipeline support, LoRA loading. (HuggingFace, well-maintained.)
- **`transformers`** — required by diffusers for text encoders.
- **`torch`** — PyTorch with CUDA 12.x. Match the RunPod image's CUDA version.
- **`accelerate`** — used by diffusers for device management.
- **`safetensors`** — LoRA file loading.

### Upscalers
- **`spandrel`** — clean loader for many upscaler model formats (Real-ESRGAN, SwinIR, etc.). Single dependency instead of two separate packages.
- Models downloaded on first run, cached.

### Image I/O
- **`Pillow`** — image load/save.
- **`numpy`** — interop with tensors.

### CLI
- **`argparse`** — stdlib, no external dep.

### Optional / deferred
- `insightface` — face detection for v2 face pass.
- `wandb` or similar — for A/B testing later.

## Models to be Downloaded on First Run

- **Flux.1-dev** — from HuggingFace. Requires HF token with Flux license accepted. ~24GB download.
- **Real-ESRGAN** weights — `RealESRGAN_x4plus.pth` or similar, downscale to target.
- **SwinIR** weights — pretrained x4 model.

Cache directory: `/workspace/.cache/flux-upscale/` (configurable via env var).

---

## RunPod Setup (in README)

1. Spin up an H100/H200/B200 pod with a PyTorch 2.x + CUDA 12.x image.
2. Set `HF_TOKEN` env var (for Flux.1-dev gated access).
3. `git clone <repo>` into `/workspace`.
4. `pip install -r requirements.txt`.
5. First run downloads models (~25GB total). Subsequent runs are fast.
6. Run upscale command.

## Default Prompt (when `--prompt` not provided)

```
preserve exact identity, preserve facial structure, preserve expression,
preserve composition, natural skin texture, realistic fine detail,
no facial reshaping, no plastic skin, no beauty filter
```

If the user provides `--prompt`, it replaces this entirely (so they can include their LoRA's trigger token + any subject-specific cues).

A future improvement: split into `--trigger` (prepended) and `--prompt` (the restraint phrases) so users don't have to remember the boilerplate.

## Known Unknowns / Things to Test After v1 Works

1. Whether the **0.3–0.6 LoRA scale range** is actually right for ai-toolkit LoRAs specifically. May need to be higher or lower depending on how ai-toolkit weights its training.
2. Whether SwinIR or Real-ESRGAN is genuinely better as a base for Flux refinement. Worth a head-to-head once v1 runs.
3. Whether **tiled refinement** is needed at 3x or higher scales. (For 2x at H100 VRAM, probably not. For 4x+ later, likely yes.)
4. Whether the **face-only pass** materially improves output, or whether full-frame refinement at H100 resolution is already enough.
5. Whether **prompt content** matters much at low denoise. Hypothesis: at 0.10 denoise, the prompt barely steers anything. Worth testing with empty prompt as a baseline.
